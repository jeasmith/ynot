# Official YNAB API constraints for persistence-free tagging

Checked 2026-07-13 against YNAB's official API documentation and current OpenAPI 3.1.1 document (API v1.86.0).

## Answer

Ynot can be a browser-only, single-user application that keeps transaction and tag data exclusively in YNAB **for transaction-level tags**. YNAB supports direct browser requests, personal access tokens, reading parent and split memos, single-transaction updates, bulk transaction updates, and delta reads. A parent transaction memo can therefore be read, transformed, and written without a Ynot database. [YNAB API documentation](https://api.ynab.com/) [Official JavaScript SDK: browser support](https://github.com/ynab/ynab-sdk-js#browser) [Current OpenAPI document](https://api.ynab.com/papi/open_api_spec.yaml)

The currently stated destination is not fully implementable through the supported API, however. The API returns memos on existing subtransactions, but explicitly says that updating `subtransactions` on an existing split transaction is unsupported, and it exposes no standalone subtransaction-update endpoint. Ynot therefore cannot write tags into the memo of an existing split. A follow-up product decision must either encode split associations in the parent memo, remove split-level tagging from the first version, or change the persistence/system boundary. [Current OpenAPI document: `SaveTransactionWithOptionalFields`, `SaveSubTransaction`, and transaction paths](https://api.ynab.com/papi/open_api_spec.yaml)

The API also has no documented compare-and-set mechanism for memo writes. Transaction update requests have no ETag, `If-Match`, revision, transaction version, or `last_knowledge_of_server` precondition. Ynot can reduce conflicts by re-reading and comparing the memo immediately before an update, then reconciling after it, but it cannot guarantee that another edit will not land between the read and write. [Current OpenAPI document: transaction update operations and schemas](https://api.ynab.com/papi/open_api_spec.yaml)

## Authentication and browser execution

- Every API request requires an access token sent as an HTTP Bearer token over HTTPS. YNAB supports personal access tokens and OAuth applications. [YNAB API documentation: Authentication and Access Token Usage](https://api.ynab.com/)
- A personal access token is explicitly intended for an account owner accessing their own account, does not expire, can be revoked, cannot be retrieved again after creation, and must be stored safely and not shared. That matches Ynot's single-user scope and avoids an OAuth flow. [YNAB API documentation: Personal Access Tokens](https://api.ynab.com/)
- OAuth is required when an application obtains access for other users. YNAB documents an implicit/client-side flow whose token expires after two hours and has no refresh token, plus an authorization-code/server-side flow with refresh tokens; authorization-code requests can use `state` and PKCE. A read-only OAuth scope cannot call write endpoints, so Ynot would need write-capable authorization if it later adopted OAuth. [YNAB API documentation: OAuth Applications](https://api.ynab.com/)
- YNAB supports CORS for direct web-application calls; its official JavaScript SDK states that asynchronous browser requests are supported from any origin. A server proxy is therefore not required by the API merely to read or update transactions. [YNAB API documentation: Usage](https://api.ynab.com/) [Official JavaScript SDK: Browser](https://github.com/ynab/ynab-sdk-js#browser)
- YNAB requires access tokens to be handled securely and never exposed to a third party, but its documentation does not prescribe a particular persistent browser-storage mechanism. The execution-and-credential decision must define what “securely on this device” means for the deployed app. [YNAB API Terms of Service](https://api.ynab.com/)

## Plans, transactions, splits, and memos

- Since API v1.79.0, the documented resource name is `plan` and the primary paths use `/plans/{plan_id}`; legacy `/budgets/{budget_id}` paths remain for backward compatibility but are no longer documented. Product language may still say “budget,” but implementation should use the current `plans` API. [YNAB API changelog: v1.79.0](https://api.ynab.com/)
- `GET /plans` lists accessible plans. `GET /plans/{plan_id}/transactions` returns plan transactions but excludes pending transactions. The transaction list supports `since_date`, `until_date`, the limited filters `uncategorized` and `unapproved`, and `last_knowledge_of_server`; it defines no pagination parameters. [Current OpenAPI document: `GET /plans/{plan_id}/transactions`](https://api.ynab.com/papi/open_api_spec.yaml)
- Since API v1.85.0, omitting `since_date` defaults transaction listings to one year ago. Ynot must pass an explicit old-enough `since_date`—potentially based on the plan's `first_month`—to derive the complete tag vocabulary and totals from older transactions. [YNAB API changelog: v1.85.0](https://api.ynab.com/)
- A transaction response includes a stable transaction `id`, date, signed amount, nullable memo, account, payee/category identifiers, transfer identifiers where applicable, deleted state, and related display fields. A detailed split transaction contains a `subtransactions` array; each subtransaction has its own `id`, parent `transaction_id`, signed amount, nullable memo, category/payee data, transfer data, and deleted state. [Current OpenAPI document: `TransactionSummaryBase`, `TransactionDetail`, and `SubTransactionBase`](https://api.ynab.com/papi/open_api_spec.yaml)
- Transaction and subtransaction memos are both nullable. On writes, each memo has a maximum length of 500 characters. Tag syntax and any split-association encoding must share that fixed capacity with the user's ordinary memo text. [Current OpenAPI document: `SaveTransactionWithOptionalFields` and `SaveSubTransaction`](https://api.ynab.com/papi/open_api_spec.yaml)
- Scheduled transactions have separate endpoints and schemas. The ordinary transaction listing already excludes pending transactions; the first-version exclusion of scheduled transactions is compatible with the API surface. [Current OpenAPI document: transaction and scheduled-transaction paths](https://api.ynab.com/papi/open_api_spec.yaml)

## Read, update, and bulk-write operations

- `GET /plans/{plan_id}/transactions/{transaction_id}` reads a single transaction. `PUT` on that resource updates one transaction, while `PATCH /plans/{plan_id}/transactions` updates multiple transactions identified by `id` or `import_id`. The save schema's fields are optional, so a memo-only update can identify the transaction and provide its new memo without deliberately changing amount, category, approval, or other fields. [Current OpenAPI document: transaction paths, `PutTransactionWrapper`, and `PatchTransactionsWrapper`](https://api.ynab.com/papi/open_api_spec.yaml)
- The bulk update endpoint accepts an array and returns saved transaction IDs, saved transaction representations, and updated `server_knowledge`; the current specification documents a `209` success response. It does **not** document a maximum array size, all-or-nothing atomicity, rollback behavior, or a structured per-item failure contract. Bulk operations must therefore be designed as potentially uncertain until their results are reconciled with a fresh read. [Current OpenAPI document: `PATCH /plans/{plan_id}/transactions` and `SaveTransactionsResponse`](https://api.ynab.com/papi/open_api_spec.yaml)
- Parent transaction memos are supported update fields, so transaction-level apply/remove and global rename/delete are API-feasible. The 500-character validation limit still applies to every rewritten memo. [Current OpenAPI document: `SaveTransactionWithOptionalFields`](https://api.ynab.com/papi/open_api_spec.yaml)
- Subtransaction memos appear in read responses and can be supplied when creating a split, but “Updating `subtransactions` on an existing split transaction is not supported.” There is no documented `/subtransactions/{id}` update route. Existing split-level memo tagging is therefore unavailable through the supported API. [Current OpenAPI document: `SaveTransactionWithOptionalFields`, `SaveSubTransaction`, and all paths](https://api.ynab.com/papi/open_api_spec.yaml)

## Delta reads and a no-persistence runtime

- The plan transaction list supports delta requests. A full response includes `server_knowledge`; a later request can send that number as `last_knowledge_of_server` and receive only entities changed since that knowledge point. Deleted transactions and deleted subtransactions are represented in delta data with `deleted: true`. [YNAB API documentation: Delta Requests](https://api.ynab.com/) [Current OpenAPI document: `TransactionsResponse`, `TransactionSummaryBase`, and `SubTransactionBase`](https://api.ynab.com/papi/open_api_spec.yaml)
- `server_knowledge` is a read cursor, not a write precondition. It is returned by save responses but is not accepted by either single or bulk transaction update operations. It cannot prevent stale writes. [Current OpenAPI document: transaction paths, `TransactionsResponse`, `TransactionResponse`, and `SaveTransactionsResponse`](https://api.ynab.com/papi/open_api_spec.yaml)
- Under Ynot's no-persistence rule, transaction state and `server_knowledge` can exist only in memory for the live browser session. Reloading or opening a new session requires another explicit full-history read; subsequent refreshes in that same session can use deltas and merge changed/deleted records into memory. This is an application consequence of YNAB's cursor model and the chosen persistence boundary. [YNAB API documentation: Delta Requests](https://api.ynab.com/)

## Rate and failure constraints

- Each access token is limited to 200 requests per hour in a rolling window; exceeding the limit returns HTTP 429. YNAB recommends caching and delta requests to reduce traffic. In Ynot, that cache must be session-only unless the persistence boundary changes. [YNAB API documentation: Best Practices, Delta Requests, and Rate Limiting](https://api.ynab.com/)
- The documented error model also includes `403 data_limit_reached`, transient 500/503 responses, and a 503 timeout when a request takes longer than 30 seconds. Because the transaction list is unpaginated but supports date bounds, a robust initial load may need explicit date windows rather than assuming one unlimited full-history response. [YNAB API documentation: Errors](https://api.ynab.com/) [Current OpenAPI document: transaction-list parameters](https://api.ynab.com/papi/open_api_spec.yaml)
- The generic error table mentions HTTP 409 for a conflicting PUT or POST, but the transaction-specific specification documents 409 only for creating a transaction whose `import_id` already exists. Single update documents 200/400; bulk update documents 209/400. There is no documented stale-memo conflict response. [YNAB API documentation: Errors](https://api.ynab.com/) [Current OpenAPI document: transaction update operations](https://api.ynab.com/papi/open_api_spec.yaml)

## Consequences for the Wayfinder map

1. **Parent-only tagging is technically viable without Ynot data persistence.** Direct browser calls, PAT authentication, parent memo reads/writes, and session-only state are all supported. [YNAB API documentation](https://api.ynab.com/) [Current OpenAPI document](https://api.ynab.com/papi/open_api_spec.yaml)
2. **Existing split-level tagging is the blocking mismatch.** The map must decide whether split tags are encoded in the parent memo, dropped from the first version, or enabled by changing a standing boundary. Parent encoding would consume the parent's 500-character memo allowance and needs an identity/grammar decision. [Current OpenAPI document: memo and subtransaction update schemas](https://api.ynab.com/papi/open_api_spec.yaml)
3. **“Prevent silent overwrite” cannot be an absolute guarantee with this API.** A pre-write GET/compare, memo-only update, and post-write reconciliation are prudent, but there remains an unavoidable race without conditional writes. The safe-write ticket must define and disclose this best-effort boundary. [Current OpenAPI document: transaction operations](https://api.ynab.com/papi/open_api_spec.yaml)
4. **Every fresh session must reconstruct its complete view.** The initial request must override the one-year default, then session refreshes can use `server_knowledge`. Date-windowing, loading/progress states, and rate-limit handling belong in the execution and acceptance decisions. [YNAB API changelog: v1.85.0](https://api.ynab.com/) [YNAB API documentation: Delta Requests and Rate Limiting](https://api.ynab.com/)
5. **Bulk parent updates exist but have underspecified failure semantics.** Preview, bounded batches, progress, verification, and undo should operate on confirmed results rather than assuming atomic success. [Current OpenAPI document: bulk transaction update](https://api.ynab.com/papi/open_api_spec.yaml)
